Тестовое задание: Phone Validator Service

Контекст

В нашу CRM поступают лиды из разных источников: лендинги, формы обратной связи, сторонние интеграции. Одним из ключевых полей является номер телефона клиента — именно по нему менеджеры связываются с лидом.
Лиды поступают преимущественно из США, затем из Канады, Южной Америки и Европы.
Проблема в том, что номера часто приходят в некорректном виде. Пользователь может случайно продублировать цифру при вводе. Форма на сайте может не добавить код страны. Сторонняя интеграция может передать номер в произвольном формате. В итоге менеджер видит номер, пытается позвонить — и не может дозвониться.
Задача

Нужно реализовать сервис, который будет подключён к CRM через webhook. Когда новый лид попадает в CRM, она отправляет данные лида на сервис, тот валидирует номер телефона и возвращает результат обратно — CRM обновляет поле с номером на нормализованное значение.
В конечном итоге все номера в лидах должны быть в формате E.164. Сервис должен привести валидные номера к этому формату, а невалидные — вернуть с указанием причины, по которой привести их невозможно.
Критерии валидации

Подумай, какие ситуации реально мешают дозвониться до клиента, и покрой их. Несколько очевидных примеров для ориентира:


9161234567 — непонятно, это российский номер без кода страны или что-то другое

11111111111 — явно не настоящий номер

123 — слишком короткий, чтобы быть номером телефона

Предполагаемая стратегия валидации

Валидация номера предполагается в два этапа.
На первом этапе номер проходит детерминированные проверки: отсеиваются очевидно невалидные случаи — слишком короткие строки, одинаковые цифры, отсутствие кода страны и прочие аномалии, которые можно выявить без внешних сервисов.
На втором этапе номера, которые не прошли первый этап, но выглядят потенциально исправимыми, отправляются в LLM — с задачей попытаться восстановить номер до корректного E.164. Результат от модели необходимо повторно верифицировать перед тем, как вернуть в CRM.

Формат взаимодействия

Сервис должен предоставлять HTTP API, пригодный для подключения в качестве webhook. Структуру запроса и ответа определи самостоятельно, исходя из описанного сценария.
Фронтенд часть — дашборд сервиса

Помимо бэкенда, нужно реализовать SPA-дашборд, который отображает метрики работы сервиса.
Что должно быть на дашборде — на твоё усмотрение, но как минимум:

общее количество лидов, прошедших через сервис
соотношение успешных и неуспешных валидаций
распределение по причинам отклонения
динамика по времени

Дизайн выбирай самостоятельно.

Подсказки по тест-кейсам

При написании тестов подумай о двух группах сценариев.
Детерминированная валидация — номера, которые можно однозначно проверить без LLM:

номера уже в корректном E.164 (разные страны: US, UK, Мексика, Индия и т.д.)
номера с неизвестным или несуществующим кодом страны

Исправление через OpenAI — номера, которые сломаны, но потенциально восстановимы:

10-значный US-номер без кода страны
номер с задублированным кодом страны в начале
номер с форматными символами (скобки, дефисы)
номер с чужим кодом страны, ошибочно приклеенным в начале
номер без знака +


И отдельно — случаи, когда исправление невозможно в принципе: слишком короткая строка, все нули, бессмысленный текст, пустая строка. В таких случаях сервис должен явно вернуть, что номер восстановить нельзя.
